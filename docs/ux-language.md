# Venus 用户可见文案原则

Venus 的文案目标是让用户自然地离开工作界面片刻，而不是制造健康焦虑、效率压力或任务感。界面文案必须短、轻、清楚，并服务于当前状态和下一步动作。

## 基本原则

- 使用平静、个人化、低压力的短句。
- 优先描述当前状态和可执行动作，不解释产品理念。
- 避免医疗化、惩罚式、效率焦虑和打卡式表达。
- 不把休息包装成任务、绩效、挑战或健康诊断。
- 错误和不可用状态必须可恢复，语气克制，不责备用户。

## 推荐表达

| 场景 | 推荐文案 |
| --- | --- |
| 休息邀请 | 休息一下 |
| 稍后提醒 | 稍后提醒 |
| 跳过本次 | 跳过本次 |
| 开始休息 | 开始休息 |
| 结束休息 | 回到工作 |
| 内容加载 | 正在准备这一刻 |
| fallback | 先用一处安静画面 |
| 声音开启 | 开启声音 |
| 静音 | 静音 |
| 声音不可用 | 声音暂时不可用 |
| 完成休息 | 可以回来了 |

## 避免表达

| 类型 | 避免文案 | 原因 |
| --- | --- | --- |
| 医疗化 | 长时间工作会损害健康 | 制造诊断感和焦虑 |
| 惩罚式 | 你必须休息 | 剥夺用户控制感 |
| 效率焦虑 | 休息后效率更高 | 把休息变成生产力工具 |
| 打卡式 | 今日休息任务完成 | 强化任务负担 |
| 游戏化奖励 | 恭喜达成连续休息 | 不符合轻、静、美的定位 |
| 解释产品 | 这里是沉浸式美好空间 | 让界面变成说明书 |

## 状态文案规则

- Loading: 只说明正在准备，不展示技术细节。
- Empty: 给出可继续的平静状态，不让用户感到失败。
- Error: 使用可恢复表达，避免错误码直接暴露给用户。
- Success: 不使用奖励、成就或打卡语气。
- Disabled: 说明当前不可操作的直接原因，不责备用户。
- Audio unavailable: 明确声音暂时不可用，同时保持无声休息路径。

## 操作文案规则

- 主操作使用动词短句，例如“开始休息”“回到工作”。
- 次操作保持同等清楚，例如“稍后提醒”“跳过本次”。
- 控制项不得堆叠说明文字，复杂解释应放入文档而不是产品界面。
- 快速退出、静音和音量控制必须始终能被重新发现。

## 审查清单

- 文案是否能在一眼内理解当前状态。
- 文案是否避免了“必须”“应该立刻”“风险”“任务”“打卡”“效率”等压力词。
- 错误或离线状态是否仍然允许用户继续休息。
- 文案是否没有遮挡视觉主体或要求用户阅读教程。
- 文案是否与 `specs/001-beautiful-rest-space/design-direction.md` 的轻、静、美方向一致。
